# COMPLETE AGENT ARCHITECTURE
## AI Business Audit Platform
*Final version — ready to feed into Claude Code*

---

## OVERVIEW

7 agents. Sequential pipeline with Agents 5 and 6 running in parallel. Every agent has one focused job. Output of each agent becomes the input for the next. The entire pipeline runs as a background async job after audit completion — user sees a processing screen while it runs. Report renders only when Agent 7 completes successfully.

---

## THE PIPELINE

```
User completes audit session
            ↓
      Agent 1 (live, during audit)
            ↓
      Agent 2 — Business Profiler
            ↓
      Agent 3 — Gap Analyzer
            ↓
      Agent 4 — Solution Mapper (RAG runs here, per-gap loop)
            ↓
    ┌────────┴────────┐
Agent 5            Agent 6
Report Assembler   Diagram Architect (per-gap loop)
    └────────┬────────┘
            ↓
      Agent 7 — Final Compiler
            ↓
      Saved to reports table in Supabase
            ↓
      Report page renders at yourapp.com/audit/report/[id]
```

Agents 5 and 6 run in parallel after Agent 4 completes — cutting total processing time roughly in half.

## AGENT 1 — QUESTION ENGINE

### When It Runs
Live, during the audit session. Called on every single answer submission. This is the only agent that runs in real time — all other agents run in the background after the session ends.

### Job
Conduct the entire discovery conversation. Ask the right questions, generate intelligent contextual follow-ups based on each answer, capture the business's existing tool stack, detect which business functions actually exist in this company, and signal when enough signal has been collected across all relevant functions to stop.

### Critical Constraint
Every question must be engineered to surface an automation opportunity. If a line of questioning cannot lead to an automation recommendation, Agent 1 does not pursue it. This is not a general business consulting conversation — it is a targeted automation discovery session.

### Performance Optimization (Speed & Reliability)

**Use a Fast Model:**
Agent 1 uses **Claude 3.5 Haiku** (or equivalent fast model) — NOT the heavier Sonnet/Opus. Agent 1's job is relatively simple (pick the next question from a constrained taxonomy). A fast model cuts latency from ~3s to ~800ms per question generation.

**Streaming Response:**
The next question is streamed token-by-token to the UI using Server-Sent Events (SSE). The user sees the question appearing in real-time (like ChatGPT typing) — this feels instant even if total generation takes 1-2 seconds.

**Optimistic UI + Transition Animation:**
When the user clicks "Continue →":
1. Immediately show a smooth typing animation / skeleton on the next question card
2. The answer gets sent to the API in the background
3. By the time the animation finishes (~500ms), the streamed response is usually already arriving
4. Zero perceived wait time for the user

**Keep the Prompt Lean (Sliding Window):**
Agent 1 does NOT receive the full conversation history on every call. Instead:
- Last 3-4 Q&A pairs in full (recent context)
- A compressed rolling summary of all earlier Q&A (generated by the API route, not the LLM)
- Business context + current coverage status object
- This keeps token count low → faster response → cheaper per call

### Business Functions (The 8 Possible Categories)
Agent 1 does not assume all 8 exist in every business. It detects which are relevant based on company size, team structure, and early answers.

```
1. Sales & Lead Generation
2. Customer Onboarding & Communication
3. Operations & Project Management
4. Finance & Administration
5. Customer Support & Retention
6. Marketing & Content
7. HR & Team Management
8. Data & Reporting
```

A 4-person team may only have 3-4 relevant functions. Agent 1 identifies this early and does not ask questions about functions that don't apply.

### Tool Stack Capture
One dedicated question in the early part of the conversation explicitly asks what tools and software the business currently uses across all areas. This answer is flagged, extracted, and passed to every downstream agent. It is the single most important data point for producing specific, actionable recommendations.

### Completion Logic
- Signals completion when sufficient depth has been reached across all detected relevant functions
- Hard maximum of 20-25 questions to prevent audit fatigue
- Short or vague answers trigger follow-up questions within the same category before moving on
- Rich, detailed answers allow moving to the next category sooner
- Never a fixed question count — completion is determined by coverage quality, not quantity

### Inputs Per Call
- Last 3-4 Q&A pairs in full + compressed summary of earlier Q&A (sliding window)
- Business context captured at session start (name, industry, company size)
- Question taxonomy (8 function categories injected as hard constraint)
- Current question count
- Coverage status across all detected functions

### Output Per Call
```json
{
  "next_question": "string — the exact question to ask next",
  "question_category": "string — which of the 8 functions this belongs to",
  "is_complete": "boolean — true when sufficient coverage achieved",
  "coverage_status": {
    "sales": "covered | partial | uncovered | not_applicable",
    "operations": "covered | partial | uncovered | not_applicable",
    "finance": "covered | partial | uncovered | not_applicable",
    "customer_support": "covered | partial | uncovered | not_applicable",
    "marketing": "covered | partial | uncovered | not_applicable",
    "hr": "covered | partial | uncovered | not_applicable",
    "data_reporting": "covered | partial | uncovered | not_applicable",
    "technology": "covered | partial | uncovered | not_applicable"
  },
  "detected_tool_stack": ["array of tools mentioned so far — updates with each call"]
}
```

## AGENT 2 — BUSINESS PROFILER

### When It Runs
First agent in the background pipeline. Runs immediately after audit session is marked complete.

### Model
**Claude Sonnet** — this agent needs strong analytical reasoning to build a rich business profile.

### Job
Transform the raw Q&A transcript into a comprehensive, deeply structured business profile. This is not a summary. This is a detailed picture of how the business actually operates — rich enough that someone who has never spoken to this business could read it and fully understand their operations, team structure, daily workflows, where value is created, and where time is lost.

Think of it as the brief a senior consultant writes before walking into a client engagement. Every downstream agent reads this profile as their primary source of truth about the business.

### Dynamic Function Detection
Agent 2 does not assume all 8 functions exist. It first detects which functions are actually present in this business based on the transcript, then only profiles those. Functions that don't exist are explicitly marked as not applicable with a reason. This prevents irrelevant sections appearing in the report.

### Inputs
- Complete Q&A transcript (all question/answer pairs with categories and order)
- Business name, industry, company size from session metadata
- Detected tool stack from Agent 1's final coverage_status output

## AGENT 3 — GAP ANALYZER

### When It Runs
After Agent 2 completes. Receives a trimmed version of Agent 2's output (see Token Optimization section below).

### Model
**Claude Sonnet** — needs strong analytical reasoning for gap identification and scoring.

### Job
Identify every automation gap across all detected business functions. Score each gap by severity and automation potential. Calculate the real time and cost impact of each gap. Assign maturity scores to each detected function. Rank all gaps by priority. This agent's output is the analytical core of the entire report.

### What Counts As An Automation Gap
Any process or task in the business that is currently:
- Done manually by a human when it could be triggered and executed automatically
- Repeated on a schedule when it could be event-driven
- Dependent on a person remembering to do it when it could be system-driven
- Causing delays, errors, or inconsistency because humans are in the loop
- Consuming significant time from high-value team members for low-value repetitive work

### Maturity Scoring
Each detected function is scored 0-100 based on:
- How many manual processes exist within it
- Severity of those manual processes
- Size and seniority of team affected
- Volume of the manual work
- Availability of automation solutions for the identified gaps

Score is benchmarked against generic industry baselines:
- **Small business (1-10 people):** baseline 35/100
- **Mid-market (11-50 people):** baseline 50/100
- **Enterprise (50+ people):** baseline 65/100

These baselines are soft estimates. As the platform collects real audit data over time, they can be replaced with actual industry averages calculated from the database. Status is assigned: below average / on par / above average.

### Inputs (Trimmed from Agent 2)
- `business_profile.key_business_context` from Agent 2
- `function_profiles` from Agent 2 (only the detected functions)
- `detected_functions` list from Agent 2
- `current_tool_stack` from Agent 2
- `cross_function_observations` from Agent 2
- Company size for benchmark selection

**NOT passed:** `zoomed_out_business_view` (redundant with `key_business_context` for Agent 3's job)

### Output
```json
{
  "overall_maturity_score": "number 0-100",
  "function_scores": {
    "sales": {
      "score": "number 0-100",
      "industry_average": "number",

      "score_rationale": "string — why this function received this score"
    }
  },
  "gaps": [
    {
      "gap_id": "uuid",
      "gap_name": "string — clear descriptive name",
      "business_function": "string — which of the 8 functions",
      "current_situation": "string — detailed description of exactly what is happening manually right now, who is doing it, how often, what triggers it, what the output is, and what happens next",
      "affected_team_members": "string — who specifically is spending time on this",
      "time_cost_per_week_hours": "number — realistic estimate based on answers",
      "estimated_annual_cost": "number — calculated from time cost and assumed hourly rate",
      "severity": "high | medium | low",
      "automation_potential": "high | medium | low",
```

## AGENT 4 — SOLUTION MAPPER

### When It Runs
After Agent 3 completes. This is where RAG runs.

### Model
**Claude Sonnet** — needs strong reasoning for solution design and ROI calculation.

### Job
For every gap identified by Agent 3, find and design the most specific, actionable automation solution possible. Use the business's existing tool stack as the primary lens — always try to automate within what they already have before recommending new tools. Pull the most relevant case studies from the database via semantic search. Calculate precise ROI per solution.

### Per-Gap Looping (Critical Optimization)
Agent 4 does NOT process all gaps in a single massive prompt. Instead:
```
For each gap in Agent 3's output:
    1. Generate embedding of gap description + business context
    2. Semantic search case_studies table using pgvector → top 3 results
    3. Call Claude with: this ONE gap + tool stack + case studies + relevant function profile
    4. Receive solution for this ONE gap
    5. Append to solutions array
```
**Why per-gap looping:**
- Dramatically smaller context per call → faster, cheaper, more reliable
- Each solution gets the LLM's full attention (no "fatigue" on gap #8 of 12)
- Individual gap calls can be parallelized (run 3-5 concurrently with Promise.all)
- If one gap's solution fails, others aren't affected

### RAG Process Per Gap
```
1. Generate embedding of gap description + business context
2. Semantic search case_studies table using pgvector
3. Search order: agency's own case studies first → platform-wide case studies second
4. Return top 3 most similar case studies
5. Pass to Claude: gap + tool stack + case studies + relevant function profile
6. Generate specific solution tailored to this exact business
```

### Case Studies Strategy

**Phase 1 — Platform Seed (Build Now):**
- Seed `case_studies` table with ~20-30 generic automation case studies
- Common scenarios: "automated lead follow-up with HubSpot + Zapier", "invoice processing with QuickBooks + Make.com", "customer onboarding email sequences", etc.
- Each case study gets a pgvector embedding for semantic search
- Agent 4 uses these as baseline RAG context

**Phase 2 — Agency Uploads (Agency Mode Feature):**
- Agency owners upload their own case studies through a dedicated page in Settings
- System generates embedding and stores with agency's `workspace_id`
- Agent 4's RAG query searches: **agency's own case studies first → platform-wide second**
- This makes each agency's reports smarter and more personalized over time
- **Retention moat:** the more case studies an agency uploads, the better their reports get, making it harder to leave the platform

### Tool Stack Priority Logic
```
Priority 1: Can this be solved with tools they already have?
            → Recommend automation within existing stack first

Priority 2: Can this be solved by connecting two tools they already have?
            → Recommend integration between existing tools

Priority 3: Is there a new tool that integrates with their existing stack?
            → Recommend new tool that slots into what they have

Priority 4: Does this require a fully new tool or stack?
            → Only recommend when no existing tool option exists
```

### Inputs Per Gap Call
- ONE gap from Agent 3 with full detail
- `current_tool_stack` from Agent 2
- Relevant `function_profile` from Agent 2 (matching this gap's business_function)
- Top 3 RAG results from `case_studies` table for this specific gap
- `business_profile.key_business_context` from Agent 2

**NOT passed:** All other gaps, all other function profiles, zoomed_out_business_view (keeps context small)

### Output (Aggregated across all per-gap calls)

## AGENT 6 — DIAGRAM ARCHITECT

### When It Runs
In parallel with Agent 5, immediately after Agent 4 completes.

### Model
**Claude Sonnet** — needs precision for generating valid Mermaid syntax with business-specific detail.

### Job
Design and write complete Mermaid diagram scripts for the top automation gaps. Each diagram shows only the **proposed automated system** — the clean future state after automation is implemented. Not the old manual process. The diagram must reflect this specific business's tools, team structure, data flows, triggers, and processes — not a generic template.

### Per-Gap Looping (Same Pattern As Agent 4)
Agent 6 processes one diagram at a time, NOT all diagrams in one prompt:
```
For each gap (top gaps by priority rank):
    1. Call Claude with: this ONE gap + matching solution from Agent 4 + relevant function profile + tool stack
    2. Receive Mermaid script for this ONE gap
    3. Programmatic validation (see below)
    4. Append to diagrams array
```
**Why per-gap looping:**
- Mermaid syntax is fragile — LLMs produce better diagrams when focused on one at a time
- Individual diagram calls can be parallelized (run 3-4 concurrently)
- One failed diagram doesn't block the others

### What The Diagram Must Show
For each gap, the diagram is a complete system design of the proposed automation:
- What triggers the automation (event, schedule, human action)
- Every step the automation takes
- Which specific tools are involved at each step
- Where data flows between systems
- Any conditional logic or decision points
- Where human touchpoints remain (approvals, exceptions)
- The final output or outcome
- Up to 20 nodes — use as many as needed to show the complete system accurately

### Context Fed To Agent 6 Per Diagram
Agent 6 receives focused context per gap:
- ONE gap description from Agent 3
- Matching complete solution blueprint from Agent 4 (including every step of how_it_works)
- The specific tools being used (existing stack + any new tools from this solution)
- Relevant `function_profile` from Agent 2 for this gap's business function
- `business_profile.team_structure` from Agent 2

**NOT passed:** All other gaps, all other solutions, all other function profiles, full business context (keeps each call focused)

### Programmatic Mermaid Validation (Critical)
LLMs frequently produce Mermaid syntax that looks correct but has subtle errors. Agent 6 does NOT self-validate. Instead, the backend validates programmatically:

```
Agent 6 generates Mermaid script
            ↓
Backend code runs: mermaid.parse(script)  ← using mermaid npm package server-side
            ↓
    Valid? → Pass to Agent 7 ✅
    Invalid? → Send error message back to Agent 6:
              "This Mermaid script has a syntax error: [error details].
               Fix it and return only the corrected script."
            ↓
    Retry (max 2 attempts)
            ↓
    Still invalid after 2 retries? → Skip this diagram.
    Save Agent 6's diagram_narrative text as fallback.
    Report page shows narrative description instead of broken diagram.
```

**Frontend double-safety (belt and suspenders):**
```tsx
try {
  mermaid.render('diagram-id', script)
} catch {
  // Show: "Here's a walkthrough of how this automation works:"
  // Then display Agent 6's diagram_narrative text
  // User still gets full value — just text instead of visual
}
```

### Inputs Per Diagram Call
- ONE gap from Agent 3 (by priority rank)
- Matching ONE solution from Agent 4
- Relevant `function_profile` from Agent 2
- `current_tool_stack` from Agent 2
- `business_profile.team_structure` from Agent 2

### Output (Aggregated across all per-gap calls)
```json
{
  "diagrams": [
    {
      "gap_id": "string — matches Agent 3 gap_id",
      "gap_name": "string",
      "system_title": "string — descriptive title for the automated system",
      "mermaid_script": "string — complete, valid Mermaid flowchart syntax. Up to 20 nodes. Uses specific tool names from their stack. Clear node labels in plain language. Reflects the exact automation described in Agent 4's solution.",
      "is_validated": "boolean — true if passed programmatic mermaid.parse() validation",
      "node_count": "number",
      "diagram_narrative": "string — detailed plain language walkthrough of the diagram. Explains every step of the automated system. Written so a non-technical business owner reads this alongside the diagram and fully understands what is being built for them and how it works. Minimum 3-4 paragraphs. References specific tools and steps shown in the diagram. ALWAYS generated even if diagram fails validation — serves as fallback."
    }
  ]
}
```

## AGENT 7 — FINAL COMPILER

### When It Runs
After both Agent 5 and Agent 6 complete successfully.

### Model
**Claude Haiku** — Agent 7's job is mostly structural (merge + validate + compare). Does not need heavy reasoning.

### Job
Merge Agent 5's complete report content and Agent 6's diagrams into the single final report JSON. Inject section 6 (diagrams) into the correct position within the report structure. Handle comparison data if this is a repeat audit. Validate that all required sections are present and complete. Save final JSON to reports table in Supabase.

### Comparison Logic (Repeat Audits)
If the user has a previous report in the database, Agent 7 fetches it and generates the comparison data:
- Overall score change (current vs previous)
- Function score changes per category
- New gaps that didn't exist in the previous audit (business has grown, new problems emerged)
- Gaps that have been resolved since the previous audit
- ROI delivered in the period between audits
- Implementation progress since last audit

### Inputs
- Complete report content JSON from Agent 5
- Complete diagrams JSON from Agent 6
- Session metadata (audit date, business name, session ID, user ID)
- Previous report from database (if exists — queried by user_id, ordered by created_at)

## TOKEN COST OPTIMIZATION — WHAT EACH AGENT RECEIVES

The full pipeline accumulates context at each stage. To keep costs manageable and responses reliable, each agent receives only what it needs:

| Agent | Receives | Does NOT Receive |
|-------|----------|-----------------|
| **Agent 1** | Last 3-4 Q&A pairs + compressed summary of earlier Q&A + business context + coverage status | Full conversation history (uses sliding window) |
| **Agent 2** | Full Q&A transcript + session metadata + tool stack | Nothing trimmed — needs everything |
| **Agent 3** | `key_business_context` + `function_profiles` (detected only) + `tool_stack` + `cross_function_observations` + company size | `zoomed_out_business_view`, undetected function details |
| **Agent 4** | ONE gap per call + matching `function_profile` + `tool_stack` + RAG results + `key_business_context` | All other gaps, all other function profiles, zoomed_out_view |
| **Agent 5** | `business_profile` + `zoomed_out_business_view` + `function_scores` + `gaps` + `solutions` + `total_roi_summary` | Raw transcript, raw function profiles |
| **Agent 6** | ONE gap per call + matching ONE solution + matching `function_profile` + `tool_stack` + `team_structure` | All other gaps/solutions, full business profile |
| **Agent 7** | Agent 5 output + Agent 6 output + session metadata + previous report (if exists) | Nothing from Agents 2-4 directly |

## ERROR HANDLING ACROSS THE PIPELINE

```
Each agent output validated before passing to next agent
            ↓
Agent fails → automatic retry once
            ↓
Retry fails → flag specific agent, log error with context
            ↓
Partial pipeline completion handled gracefully:
  - Agent 2 fails → entire pipeline stops, session marked failed
  - Agent 3 fails → retry, then stop
  - Agent 4 fails → individual gap retry (per-gap loop isolates failures)
  - Agent 5 fails → Agent 6 still completes, Agent 7 waits for Agent 5 retry
  - Agent 6 fails → Agent 5 still completes, Agent 7 waits for Agent 6 retry
  - Agent 6 diagram invalid → programmatic retry (2x), then skip with text fallback
  - Agent 7 fails → retry with both inputs, then flag for manual review
            ↓
User never sees a broken report
User sees: "Some sections are still generating" if partial
Report updates automatically when missing sections complete
```

## KEY DECISIONS LOCKED

- **Automation-focused only** — every agent filters through the lens of automation opportunity
- **AI agencies as agency user type** — reports speak the language of automation professionals
- **Tool-specific recommendations** — solutions reference their actual existing tools first
- **Dynamic function detection** — no assumptions about company structure, only profile what exists
- **Mermaid.js for diagrams** — rendered client-side from text scripts, no external API
- **Programmatic Mermaid validation** — `mermaid.parse()` on backend, not LLM self-check
- **New system diagrams only** — Agent 6 shows the future automated state, not the old manual process
- **Up to 20 nodes per diagram** — enough to show a complete, detailed system design
- **Parallel processing** — Agents 5 and 6 run simultaneously to minimize report generation time
- **Per-gap looping** — Agents 4 and 6 process one gap at a time (smaller context, parallelizable, isolated failures)
- **Async pipeline** — entire pipeline runs as background job, user never waits on a loading screen
- **Streaming + fast model for Agent 1** — Haiku model + SSE streaming for sub-second perceived latency
- **Sliding window context for Agent 1** — last 3-4 Q&A in full + compressed summary of earlier Q&A
- **Partial failure handling** — individual agent failures don't break the entire report
- **Generic industry baselines** — simple size-based benchmarks, refined with real data over time
- **Case study phasing** — seed with generic case studies first, add agency upload feature in Phase 2
- **Agency case study priority** — RAG searches agency's own case studies first, platform-wide second
